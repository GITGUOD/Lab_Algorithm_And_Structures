Purpose of lab:How different ways of handling collisions affect performance.
The core problem:

    When you store data in an array using a hash function:
    key → hash → index

    You get:
        Best case:
            Each key goes to a unique spot.
        Real case:
            Multiple keys go to the same index.

    That is called collision

1. What are suitable minimum and maximum numbers of key-value pairs compared to array size?¨

What they mean: When should you resize your hash table?
If we have:
    - n = numbers of key-value pairs
    - m = array size
    we have a = n/m which will be the factor to resize the table
    If a gets too high:

        too many collisions
        long chains (or probing steps)
        slow performance

        If α gets too low:

        wasted memory
        unnecessary resizing
